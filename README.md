# TOTPPass

> 简洁、本地、安全的两步验证管理器

[![Chrome Extension](https://img.shields.io/badge/Chrome-Extension-4285F4?logo=google-chrome&logoColor=white)](https://github.com/cyber-mule/totppass)
[![License](https://img.shields.io/badge/License-MIT-green.svg)](LICENSE)

## 背景

在当今互联网环境中，两步验证（2FA）已成为账户安全的标配。然而，随之而来的痛点也日益凸显：

- **密码分散存储** - 各平台的验证码分散在不同 App 中，查找困难
- **更换设备丢失** - 换手机后验证器数据丢失，需要逐个重新绑定
- **云端同步风险** - 部分验证器依赖云端同步，存在隐私泄露隐患
- **跨平台困难** - 手机不在身边时，无法获取验证码

TOTPPass 旨在解决这些问题 —— **本地存储、浏览器集成、一键复制**。

灵感来源于 [LastPass](https://www.lastpass.com/)，但专注于 TOTP 验证码管理，更轻量、更专注。

## 功能特性

- ✅ **TOTP 标准支持** - 兼容 RFC 6238，支持 6 位和 8 位验证码
- ✅ **本地存储** - 所有密钥仅存储在浏览器本地，不上传云端
- ✅ **智能匹配** - 自动识别当前网站，扩展图标显示匹配数量
- ✅ **一键复制** - 点击验证码即可复制，无需手动选择
- ✅ **搜索过滤** - 快速查找已保存的密钥
- ✅ **实时倒计时** - 显示验证码剩余有效时间
- ✅ **密钥管理** - 支持添加、编辑、删除操作
- ✅ **备份与恢复** - 导出密钥到本地文件，支持导入恢复

## 安装

### 从 Release 下载安装（推荐）

1. 前往 [Releases](https://github.com/cyber-mule/totppass/releases) 页面
2. 下载最新版本的 `totppass-vX.X.X.zip`
3. 解压到任意目录
4. 打开 Chrome 浏览器，访问 `chrome://extensions/`
5. 开启右上角 **开发者模式**
6. 点击 **加载已解压的扩展程序**，选择解压后的目录

### 从源码安装（开发者模式）

1. 克隆仓库
   ```bash
   git clone git@github.com:cyber-mule/totppass.git
   cd totppass
   ```

2. 打开 Chrome 浏览器，访问 `chrome://extensions/`

3. 开启右上角 **开发者模式**

4. 点击 **加载已解压的扩展程序**，选择项目目录

5. 扩展图标将出现在浏览器工具栏

## 使用方式

### 添加密钥

1. 点击扩展图标，再点击右上角 **+** 按钮
2. 输入网站提供的 TOTP 密钥（通常为 16-32 位字母数字组合）
3. 选择验证码长度（默认 6 位）
4. 输入目标站点域名（如 `github.com`）
5. 可选填写名称便于识别
6. 点击 **保存密钥**

> 💡 扩展会自动填充当前访问的网站域名

### 获取验证码

**方式一：自动匹配**
- 访问已保存密钥的网站
- 扩展图标将显示匹配数量
- 点击扩展图标，当前网站的验证码显示在顶部

**方式二：手动查找**
- 点击扩展图标
- 在搜索框输入站点名称或域名
- 点击验证码卡片即可复制

### 编辑/删除密钥

- 将鼠标悬停在密钥卡片上
- 点击 **编辑** 图标修改信息
- 点击 **删除** 图标移除密钥

### 备份与恢复

**导出备份**
1. 点击扩展图标右上角的 **⋮** 菜单按钮
2. 选择 **导出备份**
3. 备份文件将自动下载为 JSON 格式（如 `totppass-backup-2024-01-01.json`）

**导入恢复**
1. 点击扩展图标右上角的 **⋮** 菜单按钮
2. 选择 **导入备份**
3. 选择之前导出的 JSON 备份文件
4. 选择导入方式：
   - **合并** - 将备份中的密钥添加到现有列表，跳过重复的站点
   - **覆盖** - 删除所有现有密钥，使用备份文件中的数据替换

> ⚠️ **安全提示**：备份文件包含明文密钥，请妥善保管，不要上传到公共位置！

## 安全说明

### ⚠️ 重要提示

**TOTPPass 是本地工具，安全责任在于用户自身。**

### 数据存储

- 所有密钥使用 Chrome Storage API 存储在浏览器本地
- 数据不会上传到任何服务器
- 数据不会同步到其他设备（除非启用浏览器同步功能）

### 潜在风险

| 风险 | 说明 | 建议 |
|------|------|------|
| 浏览器数据清除 | 清除浏览器数据将导致密钥丢失 | 定期导出备份文件 |
| 恶意扩展 | 其他恶意扩展可能读取存储数据 | 仅安装可信扩展 |
| 设备丢失 | 设备丢失将无法获取验证码 | 将备份文件保存到安全位置 |
| 浏览器同步 | 启用同步可能将密钥上传云端 | 谨慎使用同步功能 |
| 备份文件泄露 | 备份文件包含明文密钥 | 加密存储备份文件 |

### 最佳实践

1. **定期备份** - 定期导出备份文件，保存到安全位置
2. **保存恢复码** - 大多数网站在启用 2FA 时会提供恢复码，请妥善保存
3. **备份文件加密** - 将备份文件存储在加密容器中，或使用加密压缩
4. **定期检查** - 定期检查已保存的密钥，删除不再使用的条目
5. **敏感账户** - 对于高敏感账户（如银行、主邮箱），建议使用硬件安全密钥

### 与其他方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **TOTPPass** | 本地存储、浏览器集成、支持备份恢复 | 无云同步 |
| Google Authenticator | 官方支持、广泛兼容 | 换机麻烦、无云备份 |
| Authy | 云同步、多设备 | 需注册账号、数据存云端 |
| 1Password/Bitwarden | 密码+2FA 一体化 | 付费/需信任服务商 |
| 硬件密钥 (YubiKey) | 最高安全性 | 需购买设备、可能丢失 |

## 技术实现

- **TOTP 算法** - 基于 RFC 6238 实现，使用 HMAC-SHA1
- **Base32 解码** - 支持标准 Base32 编码的密钥
- **Chrome Extension Manifest V3** - 最新扩展标准
- **Service Worker** - 后台监控标签页变化

## 常见问题

**Q: 密钥格式是什么？**

A: 标准的 Base32 编码字符串，通常由网站在启用 2FA 时提供，格式如 `JBSWY3DPEHPK3PXP`。

**Q: 支持哪些网站？**

A: 所有支持 TOTP 标准的网站均可使用，包括 Google、GitHub、Twitter、Facebook、Microsoft 等。

**Q: 验证码多久更新一次？**

A: 每 30 秒更新一次，符合 TOTP 标准。

**Q: 密钥存储在哪里？**

A: 密钥仅存储在浏览器本地（Chrome Storage），不会上传到任何服务器。

**Q: 如何在多设备使用？**

A: 使用导出备份功能将密钥导出为 JSON 文件，然后在另一台设备使用导入备份功能恢复。

**Q: 备份文件安全吗？**

A: 备份文件包含明文密钥，请务必妥善保管。建议将备份文件存储在加密容器中。

**Q: 密钥丢失了怎么办？**

A: 如果有备份文件，使用导入功能恢复。如果没有备份，需要使用网站提供的恢复码，或联系网站客服重置 2FA。

## 发布流程

本项目使用 GitHub Actions 自动构建，发布流程如下：

### 方式一：通过 GitHub 网页发布（推荐）

1. **更新版本号** - 修改 `manifest.json` 中的 `version` 字段
2. **提交并推送** - 将代码合并到 main 分支并推送
3. **创建 Release** - 在 GitHub [Releases](https://github.com/cyber-mule/totppass/releases) 页面点击 "Draft a new release"
4. **填写信息**：
   - Tag: `v1.0.0`（与 manifest.json 版本号对应）
   - Title: `v1.0.0`
   - Description: 自定义发布说明（支持 Markdown）
5. **发布** - 点击 "Publish release"，GitHub Actions 自动构建并上传 zip 文件

### 方式二：通过命令行发布

```bash
# 1. 更新 manifest.json 版本号

# 2. 提交代码
git add manifest.json
git commit -m "chore: bump version to 1.1.0"

# 3. 创建 tag
git tag -a v1.1.0 -m "Release v1.1.0"

# 4. 推送代码和 tag
git push origin main --tags

# 5. 在 GitHub 上编辑 Release，添加发布说明
```

### 版本号规范

遵循 [语义化版本](https://semver.org/lang/zh-CN/)：
- `MAJOR.MINOR.PATCH`
- 例如：`1.0.0` → `1.0.1`（修复 bug）→ `1.1.0`（新功能）→ `2.0.0`（重大变更）
